# Назначение
Skill выполняет адаптацию Memory Bank, скопированного из другого проекта, под текущий репозиторий.

Основной триггер: `adapt-memory-bank-to-new-project`.
Поддерживаемый alias: `adapt_memory_bank_to_new_project`.
Если в локальной установке агента есть дополнительный prompt для адаптации Memory Bank, используй его только как вспомогательный источник. При конфликте приоритет у этого `SKILL.md`, `AGENTS.md` и фактов текущего проекта.

# Жёсткий контракт
- Всегда отвечай только на русском языке.
- До фразы пользователя `Утверждаю план` не исполняй WBS.
- До утверждения плана разрешено менять только:
  - `.memory_bank/*`
  - `AGENTS.md`
  - `CLAUDE.md`
  - `NESSY.md`
  - один отчётный файл в `.memory_bank/drafts/` при необходимости.
- В этом skill нельзя менять исходный код продукта вне Memory Bank.

# Обязательный процесс

## 1) Подготовка и аудит (read-first)
Перед любыми предложениями:
1. Прочитай корневую документацию проекта (`README`, `ROADMAP`, `TECHNICAL_OVERVIEW`, архитектурные схемы).
2. Осмотри структуру каталогов, ключевые сервисы, Docker/K8s/CI, `.env.example` (если есть).
3. Зафиксируй найденные окружения, режимы и интеграции:
   - `dev/test/stage/prod`
   - `internal/dual/external`
   - корпоративные URL/реестры и provider endpoints, если они есть и подтверждены;
   - task tracker, Git namespace и каналы коммуникации, если они используются.
4. Сверь найденное с `.memory_bank/*`, `AGENTS.md`, `CLAUDE.md`, `NESSY.md`; выпиши расхождения.
5. Если в скопированном Memory Bank есть `.memory_bank/examples/index.md`, прочитай его и выбери только те examples, которые реально помогают адаптировать текущий проект.

Правило работы с examples:
- `.memory_bank/examples/` — это библиотека адаптационных образцов, а не источник действующих правил проекта;
- examples нужно использовать как проектировочную библиотеку форм, когда создаёшь или существенно переписываешь рабочие разделы Memory Bank;
- examples помогают понять форму оформления task spec, current tasks board, guide, workflow, agent role doc, domain skill, provider policy, testing strategy, prompt handoff, worklog/checkpoint или review checklist;
- нельзя копировать example как готовое правило без замены placeholders, URL, команд, ролей, DoD и checks на факты нового проекта;
- если example становится реальным правилом, перенеси адаптированную версию в рабочий раздел (`guides/`, `workflows/`, `patterns/`, `agents/`, `specs/` или `.agents/skills/`) и обнови соответствующий `index.md`.
- если в нужном разделе шаблона сейчас есть только `index.md`, не считай это признаком, что раздел должен остаться пустым: после адаптации там могут появиться project-specific файлы, оформленные по подходящим примерам из `.memory_bank/examples/`.

Минимальная карта применения examples при адаптации:
- `adaptation_clarification_intake_example.md` -> при подготовке уточняющих вопросов и обработке ответов пользователя путями к файлам или текстом;
- `architecture_boundaries_guide_example.md` -> при фиксации целевых слоёв, каталогов, разрешённых зависимостей и structure drift;
- `testing_strategy_guide_example.md` -> при замене template testing placeholders на реальные команды и quality gates проекта;
- `task_spec_feature_example.md` -> при создании первых formal specs;
- `current_tasks_board_example.md` -> при настройке доски задач под task tracker проекта;
- `agent_role_doc_example.md` и `domain_skill_example.md` -> при создании зональных agents или repo-local domain skills;
- `documentation_project_workflow_example.md` -> при адаптации Memory Bank под проект, где основной результат — документы;
- `prompt_handoff_example.md` -> при настройке работы через отдельные агентные сессии;
- `worklog_checkpoint_example.md` -> при настройке resume/handoff для длинных задач;
- domain examples вроде `rag_eval_*`, `rag_langgraph_*`, `graph_topology_*`, `frontend_*`, `corporate_security_*` -> только если фактический проект имеет похожий домен или инфраструктурный контур.

## 2) Уточнения пользователя и evidence intake
После read-first аудита обязательно проведи короткий intake уточнений. Цель — снизить порог адаптации для пользователя: ему не нужно заранее идеально знать структуру Memory Bank, он может просто ответить текстом или дать путь к документу, где уже лежат правила проекта.

Правила intake:
- не задавай вопросы, на которые уже есть надёжный ответ в найденных файлах проекта;
- задавай только вопросы, которые улучшают адаптацию Memory Bank под конкретный проект, инфраструктуру, архитектуру, безопасность, проверки или командный процесс;
- группируй вопросы в один короткий блок до 7 вопросов;
- для каждого вопроса явно разрешай два формата ответа: `путь к файлу/разделу` или `ответ текстом`;
- если пользователь отвечает путём, прочитай указанный файл или релевантный раздел и извлеки факты сам;
- если пользователь отвечает текстом, зафиксируй это как user-provided project rule и перенеси в подходящий Memory Bank документ;
- если пользователь отвечает `не знаю`, `позже` или не даёт ответ, не блокируй адаптацию без необходимости: пометь пункт как `unconfirmed`, выбери безопасный placeholder и добавь follow-up в план портирования;
- если вопрос касается security, secrets, внешних API, production deploy, branch policy или compliance и ответа нет, используй безопасный default: не разрешать внешние вызовы, не выдумывать URL, не фиксировать production-команды и не создавать branch policy без подтверждения.

Рекомендуемые темы вопросов:
- где описаны архитектура, границы слоёв и правила размещения файлов;
- где описаны coding style, formatter, lint, тесты и обязательные quality gates;
- какие внутренние registry, proxy, LLM/RAG/embedding providers, observability и deploy-платформы разрешены;
- какие правила веток, commit, MR/PR, task tracker и release process использует команда;
- какие документы являются источниками истины: README, docs, wiki export, ADR, runbook, CI, infra repo;
- какие зоны проекта нельзя менять без отдельного согласования;
- нужен ли Memory Bank для кодового проекта, документного проекта или смешанного контура.

Формат блока вопросов:

```markdown
## Уточнения для точной адаптации

Можно отвечать двумя способами: дать путь к файлу/разделу, где уже есть ответ, или написать ответ прямо в чат. Если чего-то пока нет, напишите `не знаю` — я поставлю безопасный placeholder и follow-up.

1. Где описаны архитектура и правила размещения кода/документов? Можно указать путь, например `docs/architecture.md`, или кратко описать правила.
2. Какие команды считаются обязательными quality gates перед MR/PR? Можно указать `Makefile`, `justfile`, CI config или написать команды.
...
```

После ответов пользователя составь компактную таблицу intake-источников для плана:

```text
INTAKE_FACTS:
- topic: <architecture/testing/security/etc>
  source: <file path | user answer | unconfirmed>
  decision: <что будет записано в Memory Bank>
  target_mb_doc: <path>
```

Не превращай intake в длинную анкету. Если проект простой и факты очевидны из репозитория, достаточно 2-3 вопросов или одного вопроса о недостающих правилах.

## 3) Предложение пути
Одним сообщением:
1. Кратко перечисли, какие факты уже найдены автоматически и какие получены из ответов пользователя.
2. Перечисли unresolved пункты, если они остались, и безопасный default для каждого.
3. Предложи:
   - `slug` нового проекта (`kebab-case`)
   - ветку `cdx_memory_bank_port_<slug>`
   - путь плана `/.memory_bank/drafts/mb_port_<slug>.md`
4. Дождись подтверждения пути (`ОК путь`).

## 4) План портирования (до approve)
После `ОК путь`:
1. Сформируй WBS-план и выведи его в чат.
2. Запиши план в `/.memory_bank/drafts/mb_port_<slug>.md`.
3. Включи в план:
   - аудит артефактов старого проекта;
   - карту `find -> replace`;
   - таблицу intake-источников: `topic -> source -> decision -> target_mb_doc -> confidence`;
   - таблицу правок по файлам с DoD;
   - карту расширения разделов Memory Bank: какие папки останутся только с `index.md`, а где появятся project-specific guides/workflows/specs/agents/patterns/prompts/worklogs;
   - список использованных `.memory_bank/examples/` и решение: `использовать как форму`, `адаптировать в рабочий раздел`, `не подходит`;
   - решения по субагентам;
   - требования к frontmatter (`language: ru`, `locale: ru-RU`);
   - блок безопасности (provider policy, secrets, запрет внешних API без явного решения);
   - чек-лист финального DoD.
4. Явно дождись команды `Утверждаю план`.

## 5) Исполнение (только после approve)
После `Утверждаю план`:
1. Выполняй WBS малыми шагами.
2. После каждого шага:
   - короткий отчёт в чат;
   - обновление `mb_port_<slug>.md`.
3. Не выходи за границы заявленного scope.

## 6) Финал
В финале обязательно выдай:
1. Сводную таблицу изменений (`файл -> действие -> статус`).
2. Проверку DoD по пунктам.
3. Предложение Conventional Commit сообщений.
4. Следующий шаг (например, отдельный сеанс на разработку).

# Формат ответа
- Блок 1: `Уточнения для точной адаптации`.
- Блок 2: `Intake summary` (`topic`, `source`, `decision`, `target_mb_doc`, `confidence`).
- Блок 3: `Предложение` (`slug`, `ветка`, `путь`).
- Блок 4: `План портирования (WBS) + DoD` (после `ОК путь`).
- Блок 5: `Пошаговое исполнение и итоговый отчёт` (после `Утверждаю план`).

# Quality gate
Перед финальным ответом проверь:
1. Нет ли изменений вне разрешённых файлов.
2. Удалены ли следы старого проекта: имена, URL, task tracker, каналы коммуникации, alias, логотипы, скриншоты и доменные правила.
3. Согласованы ли `AGENTS.md`, `CLAUDE.md`, `NESSY.md` и `.memory_bank/README.md`.
4. Актуальны ли подтверждённые provider URL, registry и security rules в `providers.*.yaml` и `tech_stack.md`.
5. Если использовались `.memory_bank/examples/`, перенесены ли только адаптированные версии в рабочие разделы и не остались ли examples как process-truth.
6. Все user-provided правила и ссылки на файлы из intake либо перенесены в Memory Bank, либо явно отмечены как unresolved/follow-up.
7. Присутствует ли отчёт `mb_port_<slug>.md` с перечнем сделанных правок.
